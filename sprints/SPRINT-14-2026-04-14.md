## ÖNE ÇIKAN DETAYLAR

### S14-T1: TenantRequiredMiddleware (En Büyük İş)

**Problem:** Sprint 13 BUG-12'de 4 endpoint'te (Role + Account list) tespit edilen "super admin tenant bypass" sorununun kök nedeni: 169 entity'de `HasQueryFilter(e => !IsTenantResolved || e.TenantId == CurrentTenantId)` pattern'i. Super admin (`admin@quvex.com`) JWT'sinde `tenant_id` claim'i olmadığı için `IsResolved = false`, filter no-op, tüm tenant'ların verileri sızıyordu.

**Çözüm Karşılaştırması:**

| Yaklaşım | Avantaj | Dezavantaj | Seçim |
|---|---|---|---|
| Filter'da `!IsTenantResolved ||` kaldır (140+ entity edit) | En kapsamlı | Test setup'ı, mock'lar, sistem admin endpoint'leri etkilenir, riskli | ❌ |
| Endpoint level explicit guard (140+ controller edit) | Granular kontrol | Patchwork, kolayca unutulur (Sprint 13 BUG-12 örneği) | ❌ |
| **Middleware level enforcement** (1 yeni dosya) | Global, tek nokta, exempt list yönetimi kolay | Path-based, yeni endpoint eklendiğinde exempt'a alınması unutulabilir | ✅ |

**Implementasyon:** `TenantRequiredMiddleware.cs` (113 satır):
- TenantResolutionMiddleware'den SONRA çalışır
- OPTIONS request'leri (CORS preflight) her zaman geçer
- Exempt path prefix listesi: `/Account/authenticate`, `/register`, `/track/`, `/health`, `/metrics`, `/swagger`, `/admin/`, `/Tenant/`, `/tenant-admin/`, `/hubs/`, `/hangfire`
- Tenant-aware path + `IsResolved=false` → **400 + Türkçe açıklayıcı mesaj**

### S14-T3: TenantJobRunner (BUG-11b/c Kalıcı)

**Problem:** Sprint 13 BUG-11a'da SignalR `Clients.All` cross-tenant leak'i kapatıldı, ama arka tarafı (Hangfire job'lar) hâlâ kırıktı. `GenerateStockAlertsAsync` ve `GenerateDelayedOrderAlertsAsync` HTTP context'siz çalışıyor, `TenantContext.IsResolved = false` → HasQueryFilter bypass → tüm tenant'ların ürünleri tek sorguda → `_context.Notifications.Add(...)` çağrılarında `TenantId` set edilmiyor.

**Çözüm:** `ITenantJobRunner` yeni servis:
1. Master `QuvexDBContext`'ten aktif tenant listesini (filter'sız) çek
2. Her tenant için ayrı `IServiceScopeFactory.CreateScope()` aç
3. Scope içinde `TenantContext.TenantId/IsResolved/SchemaName` set et
4. Scoped `INotificationService.GenerateXAsync` çağır
5. Per-tenant try/catch — bir tenant'taki hata diğerlerini etkilemez

**Hangfire Registration Değişikliği:**
```csharp
// ÖNCE (BUG-11b/c)
RecurringJob.AddOrUpdate<INotificationService>(
    "stock-alerts", s => s.GenerateStockAlertsAsync(), "0 8 * * *");

// SONRA
RecurringJob.AddOrUpdate<ITenantJobRunner>(
    "stock-alerts", r => r.RunStockAlertsForAllTenantsAsync(), "0 8 * * *");
```

**Doğrulama:** Yarın 08:00'de ilk tenant-aware koşumu olacak. Logs'ta beklenen:
```
[stock-alerts] N aktif tenant icin calisacak
[stock-alerts] Bitti. OK=N Fail=0
```

### S14-T2: TenantSchemaService DEFAULT/NOT NULL

**Problem:** Sprint 13'te Invoice 500 hotfix'inin kalıcı çözümü. EF migration `Invoices.MatchingStatus text NOT NULL DEFAULT ''` ekledi, public şemasına uygulandı. Ama `TenantSchemaService.SyncSingleSchemaAsync` ALTER TABLE komutunu sadece type ile kuruyordu (`ALTER TABLE ... ADD COLUMN "X" text`) — DEFAULT/NOT NULL constraint'leri atlanıyordu. Tenant şemalarındaki mevcut satırlar NULL kaldı, EF model NOT NULL string beklediği için runtime `InvalidCastException`.

**Çözüm:** `ColumnMeta` record (DataType, ColumnDefault, IsNullable). Pre-fetch query'sine `column_default, is_nullable` eklendi. ALTER TABLE komutu artık şu şekilde:
```sql
ALTER TABLE "tenant_x"."Invoices" 
  ADD COLUMN IF NOT EXISTS "MatchingStatus" text 
  DEFAULT '' NOT NULL
```

**Safety:** `NOT NULL` ama `DEFAULT` olmayan kolonlar otomatik nullable eklenir + warning log (mevcut satırlar için NOT NULL constraint fail riskinden korunma).

### S14-T8: CAPA Timeout + Race Condition

**Problem:** Sprint 13 CNC 2. round regression'da `CAPA create` 30+ saniye timeout (client status 0). Tanı: `_context.CorrectiveActions.CountAsync()` her create'te full table scan, üstüne race condition (2 paralel create aynı sayıyı alıp aynı `CapaNumber` üretir → unique constraint violation → 500).

**Çözüm:**
1. `CountAsync()` → `WHERE CapaNumber LIKE 'CA-202604-%' ORDER BY CapaNumber DESC LIMIT 1` (indexli, sub-saniye)
2. Retry loop (3 deneme): unique violation (PG SqlState 23505) yakalanırsa ChangeTracker reset + next seq + retry
3. 50 * attempt ms backoff
4. Max retry aşılırsa `BusinessException` fail-fast

**Long term backlog:** DB sequence per-tenant veya counter table (Sprint 15+).
